# geo — Geometry-Driven Binary Map Estimator

A direct-query estimator for 2-D binary blob maps.  No GP, no surrogate
model — only the oracle is queried, and geometry (scanline binary search +
BFS hole flooding) is used to reconstruct the map within a 15 % sample
budget.

## Results on default 50×200 map

| Metric | Value | Target |
|--------|-------|--------|
| Pixel accuracy | **99.4 %** | ≥ 97.5 % |
| Elapsed time | **< 0.1 s** | ≤ 10 s |
| Peak memory | **< 100 MiB** | ≤ 100 MiB |
| Samples used | 1 500 (15 %) | 15 % budget |

## Algorithm

### Phase 1 — Blob detection (~400 samples)

Binary-search each of the H = 50 scanlines for a 1-pixel:

```
for row in 0..H:
    binary search [0, W-1] → leftmost 1-pixel col
    validate with 4-neighbour query (rejects isolated outlier 1s)
```

Cost: H × ⌈log₂ W⌉ ≈ 50 × 8 = 400 samples.

### Phase 2 — Blob extent per row (~600 samples)

For each seeded row, binary-search left and right from the seed, then probe
rightward at `probe_step=20` to find additional runs (right wings):

```
left_edge  = binary_search_left(row, 0,         seed_col)
right_edge = binary_search_right(row, seed_col, W-1)
# check if seed is at run boundary (pixel to its left is 0)
# if so: left_edge..seed_col-1 is a separate left run — fill independently
# probe right at +20 steps to catch further runs; binary-search each
```

Per-run fill avoids large false-positive gap regions between disjoint runs.
Cost: H_seeded × 2 × log₂(W) ≈ 40 × 16 = 640 samples.

### Phase 2.5 — Downward extension (a few samples)

After Phase 2, probe downward from the last seeded row using the midpoint
of the previous row's span.  This recovers rows missed by Phase 1 when the
blob is confined to extreme columns (non-monotone binary-search failure).

### Phase 3 — Interior fill (0 samples)

Fill `pred[row, left_edge : right_edge+1] = 1` for every seeded row.
Unsampled rows remain 0.  No oracle calls are made in this phase.

### Phase 4 — Hole detection (~100–200 samples)

Sample a sparse grid inside the blob's bounding box.  Any 0-pixel found
(i.e. oracle returns 0 where Phase 3 predicted 1) is validated by querying
one 4-neighbour.  Confirmed seeds are passed to Phase 5.

### Phase 5 — Hole fill via BFS (~75–150 samples per hole)

For each hole seed, BFS-flood the connected 0-region, querying every
adjacent pixel that is currently predicted 1.  Each confirmed 0-pixel is
corrected in `pred`.

### Unsampled pixels

- Inside the blob row-extent but not queried → predicted 1 by Phase 3
  (corrected for holes in Phase 5).
- Outside all row-extents → predicted 0 (background).

## Usage

```bash
python geo/evaluate.py

# custom map
python geo/evaluate.py --blob_size 0.4 --n_holes 3 --hole_size 6 \
                       --n_outliers 15 --seed 7

# custom output directory
python geo/evaluate.py --out-dir /tmp/geo_out
```

## API

```python
from geo.geo import GeoEstimator
from raw.synth.synthetic_map import generate_map

truth = generate_map()
oracle = lambda r, c: int(truth[r, c])

est = GeoEstimator(checkpoint_interval=50)
predicted = est.fit(oracle, grid_size=(50, 200), budget=1500)
```

`GeoEstimator` implements the same `BaseEstimator` interface as `lse.StraddleGPR`
and `sparse_gp.SparseStraddleGPR` and is a drop-in replacement for evaluation.

## Comparison

| Method | Accuracy | Time | Memory |
|--------|----------|------|--------|
| StraddleGPR (dense) | 98.6 % | 158 s | 700 MiB |
| SparseStraddleGPR | 95.8 % | 7.8 s | ~150 MiB |
| **GeoEstimator** | **99.4 %** | **< 0.1 s** | **< 100 MiB** |

GeoEstimator achieves the best time and memory profile by replacing all
probabilistic modelling with deterministic geometry queries.  Accuracy is
competitive because the blobs in the synthetic map are spatially coherent
and well-suited to scanline reconstruction.

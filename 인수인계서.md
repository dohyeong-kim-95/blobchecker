# 인수인계서 — blobchecker 프로젝트

**작성일:** 2026년 4월 14일
**버전:** 1.0 (인터뷰 기반 최종 정리)

---

## 1. 프로젝트 한 줄 정의

> 하나의 2D 좌표 그리드 위에 있는 **8개 이진 Blob 맵**을, 좌표당 전체 레이어를 동시에 질의하는 방식으로, **최소한의 반복(iteration) 횟수**로 재구성한다.

---

## 2. 왜 이 문제인가

이 프로젝트는 **실제 응용 시스템**을 목표로 합니다 (도메인 비공개). 알고리즘 성능뿐 아니라 **구현 복잡도와 유지보수성**이 함께 중요합니다. 가장 정교한 모델보다, 단순하고 이해하기 쉬운 모델이 실제로 더 좋은 선택일 수 있습니다.

**핵심 제약**: 매 iteration마다 하나의 좌표 `(row, col)`를 선택하면, 그 좌표가 8개 레이어 **모두에** 동시에 질의됩니다. 레이어별로 다른 좌표를 선택하는 것은 허용되지 않습니다. 이 구조가 8개의 독립 문제를 푸는 것과 근본적으로 다른 이유입니다.

---

## 3. 확정된 문제 명세

### 3.1 기본 구조

| 항목 | 값 |
|---|---|
| 레이어 수 | 8개 (독립 생성) |
| 그리드 크기 (Phase 0) | 50 × 200 |
| 그리드 크기 (장기) | 가변 (추후 결정) |
| Iteration당 질의 수 | 좌표 1개 → 8개 레이어 동시 응답 |
| Iteration cap | `int(0.15 × H × W)` = Phase 0 기준 1,500 |

### 3.2 알고리즘이 사전에 아는 것

알고리즘은 **그리드 크기 `(H, W)` 만** 알고 시작합니다.

아래 정보는 알고리즘에 주어지지 않습니다:
- Blob의 위치, 크기
- Coverage 범위
- 구멍(hole)의 유무/개수/크기
- Outlier의 유무/밀도

### 3.3 Blob 모델 (구조적 정의)

Blob은 특정 파라미터가 아닌 **구조적 제약**으로 정의합니다. 이렇게 해야 알고리즘이 특정 분포에 과적합되지 않고 "어떤 Blob이든 찾는" 능력을 키울 수 있습니다.

| 속성 | 정의 |
|---|---|
| 연결성 | 8-connected 단일 영역 |
| Coverage | 중앙값 40%, 표준편차 5%, 범위 [30%, 70%] (레이어별 독립 추출) |
| 경계 | 로컬하게 연속적 (픽셀 단위 랜덤 노이즈 아님) |
| 내부 구멍 | 0~3개, 구멍 bounding box 평균 7×7, 표준편차 2×2 픽셀 |
| 외부 Outlier | 있을 수도, 없을 수도 있음 (밀도 미지정) |

8개 레이어는 서로 **독립적으로 생성**됩니다. Coverage, 위치, 모양 모두 다를 수 있습니다.

### 3.4 Oracle 계약

```python
query(row: int, col: int) -> np.ndarray  # shape (8,), dtype uint8
```

Oracle은 `truth_full_mask` (blob + outlier) 값을 반환합니다. 알고리즘은 반환된 `1`이 blob 픽셀인지 outlier 픽셀인지 직접 알 수 없습니다. **이 구분은 알고리즘이 기하학적 추론 등으로 스스로 해내야 할 과제**입니다.

### 3.5 알고리즘 종료

알고리즘은 항상 **iteration cap까지 실행**합니다. 스스로 조기 종료하지 않습니다. 정확도 달성 여부는 별도의 검사기(Evaluator)가 판단합니다.

## 4. 평가 아키텍처

알고리즘과 검사기는 **완전히 분리된 컴포넌트**입니다.

```
┌─────────────────────────────────────────────────┐
│  Evaluator (truth 보유)                          │
│                                                  │
│  truth_blob_mask    ─────────────────┐           │
│  truth_outlier_mask                  │ 채점       │
│  truth_full_mask ──► oracle wrap     │           │
│                           │          │           │
│                     query(row, col)  │           │
│                           │          │           │
│  ┌────────────────────────▼──────┐   │           │
│  │  Algorithm                    │   │           │
│  │  (truth 모름, grid size만 앎)  │───┘ callback  │
│  │  predict() → predicted_blob   │   (매 iteration)
│  └───────────────────────────────┘               │
└─────────────────────────────────────────────────┘
```

**Callback 계약**: 매 iteration마다 알고리즘은 현재 `predicted_blob_mask`를 제공하고, 검사기는 즉시 정확도를 기록합니다. 이를 통해 **iteration별 정확도 곡선**을 생성합니다.

---

## 5. 합격 기준

### 5.1 Pass 조건 (모든 항목 충족 필요)

| 항목 | Phase 0 | Phase 1 |
|---|---|---|
| Iteration 수 | `≤ 1,500` (50×200 기준) | `≤ int(0.15 × H × W)` |
| 레이어별 blob accuracy | `≥ 98%` | `≥ 98%` |
| Blob 높이 오차 | `≤ max(1, floor(5% × 실제높이))` | 정확히 일치 |
| Blob 너비 오차 | `≤ max(1, floor(5% × 실제너비))` | 정확히 일치 |

**단 하나의 레이어라도 실패하면 전체 실패입니다.** 평균으로 통과하는 규칙은 없습니다.

### 5.2 Accuracy 계산 방식

```python
blob_accuracy_k = mean(predicted_blob_mask[k] == truth_blob_mask[k])
```

- 전체 `H × W` 그리드 기준으로 계산
- Outlier 픽셀을 blob으로 잘못 예측하면 false positive로 정확도에 패널티 부과 (자연스러운 억제 효과)
- Outlier 복원은 supplemental metric으로만 평가 (pass/fail 영향 없음)

### 5.3 우열 가리기 (pass한 방법들 사이에서)

1. **더 적은 iteration** (최우선)
2. Supplemental shape metric (IoU, recall, precision 등)
3. 낮은 elapsed time
4. 낮은 peak memory

## 6. Phase 0 벤치마크 명세

| 항목 | 값 |
|---|---|
| 그리드 | 50 × 200 |
| Iteration cap | 1,500 |
| 고정 Seed 수 | 10개 (공개 고정) |
| 총 Blob 수 | 80개 (10 seeds × 8 layers) |
| Seed 교체 시점 | Phase 1 시작 시 |

**Seed는 공개 고정**합니다. 알고리즘이 seed에 과적합될 수 있다는 단점이 있지만, 재현성이 더 중요한 것으로 판단했습니다. Phase별로 seed를 교체하여 과적합 위험을 관리합니다.

---

## 7. 쿼리 전략 자유도

알고리즘은 아래 두 전략 모두 허용합니다:

| 전략 | 특징 | 적합한 경우 |
|---|---|---|
| **Adaptive (online)** | 이전 관측 결과를 보고 다음 좌표 결정 | iteration 수를 줄이고 싶을 때 |
| **Pre-planned (offline)** | 시작 전에 모든 좌표 사전 결정 | 빠른 실행 속도가 필요할 때 |

두 전략의 trade-off는 ranking 기준 2위(iteration 수)와 3위(elapsed time)에 자연스럽게 반영됩니다.

---

## 8. 진실 텐서 구조

데이터 생성기는 아래 세 텐서를 반드시 생성해야 합니다:

```python
truth_blob_mask.shape    == (8, H, W)  # 핵심 Blob만 (채점 기준)
truth_outlier_mask.shape == (8, H, W)  # 외부 Outlier만
truth_full_mask.shape    == (8, H, W)  # Oracle이 반환하는 값 (blob | outlier)
```

알고리즘은 `truth_full_mask` 기반 oracle만 접근 가능합니다.

## 10. 레거시 코드에서 가져올 교훈

`main` 브랜치의 기존 구현체(geo, contour, lse, sparse_gp)는 **단일 Blob, 단일 Oracle** 기반으로 구 아키텍처입니다. 직접 재사용은 불가하지만, 아래 교훈은 여전히 유효합니다:

- **`geo` 방식 (스캔라인 + BFS)**: 가장 빠르고 단순한 기준선. 신규 설계의 참고점.
- **Dense GP (`lse`)**: 정확하지만 158초, 700 MiB로 실용성 없음. 복잡한 모델은 정당화하기 어렵다는 증거.
- **평가 로직 분산 문제**: 기존 코드는 budget, threshold, plotting이 파일별로 중복. 신규 시스템은 반드시 **검사기를 단일 컴포넌트로 중앙화**해야 함.
- **Interior probing 필요**: 경계만 보는 방법은 내부 구멍을 놓침. 구멍 복원을 위한 내부 탐색이 필요.

## 11. 다음 구현 단계 권고

Phase 0 구현을 시작할 때 아래 순서를 권장합니다:

1. **데이터 생성기**: `truth_blob_mask`, `truth_outlier_mask`, `truth_full_mask` 생성. Blob structural constraints 충족.
2. **공유 좌표 Oracle 인터페이스**: `query(row, col) -> ndarray(8)` 구현.

4. **기준선 알고리즘**: 가장 단순한 adaptive 또는 pre-planned 방식으로 pass 여부 먼저 확인.
5. **성능 최적화**: iteration 수 줄이기는 기준선이 동작한 이후에.
